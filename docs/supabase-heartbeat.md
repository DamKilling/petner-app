# Supabase Heartbeat

## Purpose

This workflow keeps the Supabase Free Tier project active by sending a lightweight scheduled request to the Supabase REST API. It is intended to create real database activity without changing the iOS app code or the deployment flow.

The workflow file is located at:

```text
.github/workflows/supabase-heartbeat.yml
```

## Schedule

The workflow runs twice per week:

- Monday at 02:00 UTC
- Thursday at 02:00 UTC

It also supports manual runs through `workflow_dispatch` in GitHub Actions.

## Required GitHub Secrets

Add the following secrets in GitHub:

```text
Repository Settings -> Secrets and variables -> Actions
```

Required secrets:

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`

`SUPABASE_URL` example:

```text
https://crnxjlzvcmvblmxxlcfs.supabase.co
```

Do not commit real Supabase keys to the repository. This workflow uses the anon public key through GitHub Secrets and does not require the Supabase `service_role` key.

## Heartbeat Table

The workflow calls:

```text
GET $SUPABASE_URL/rest/v1/heartbeat?select=id&limit=1
```

Create the `heartbeat` table in the Supabase SQL Editor before enabling or manually running the workflow:

```sql
create table if not exists public.heartbeat (
  id bigint generated by default as identity primary key,
  pinged_at timestamptz not null default now()
);

insert into public.heartbeat default values;
```

## Failure Behavior

The workflow uses `curl --fail --show-error --silent`. If the Supabase request fails, the GitHub Actions job will fail so the error is visible in the Actions run log.
